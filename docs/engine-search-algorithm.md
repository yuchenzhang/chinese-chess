# 本地象棋引擎搜索算法与优化说明

本文档介绍了本象棋项目中本地引擎（Local Engine）所使用的博弈树搜索（Game Tree Search）算法的原理，以及针对性能和用户体验所做的计算量优化与时间控制机制。

## 1. 核心搜索算法原理

本地引擎基于经典的 **Minimax（极大极小值）算法**，并结合了 **Alpha-Beta 剪枝** 和 **迭代加深（Iterative Deepening）**，以在有限的时间内寻找最佳着法。

### 1.1 迭代加深 (Iterative Deepening)
引擎并不会直接从一开始就深入到用户设定的最大深度（例如 `maxDepth = 4`）。相反，它会：
1. 先搜索深度为 1 的所有可能性，找到最佳步。
2. 然后搜索深度为 2 的所有可能性。
3. 依次类推，直到达到设定的最大深度，或者触发超时。

**优势**：
- **随时可中断**：如果时间耗尽，引擎可以立即返回上一个已完成深度的最佳步，从而保证响应时间。
- **提高剪枝效率**：浅层搜索能快速找到一个“较好”的着法，这个着法在更深层搜索中作为初始猜测，能大幅提高 Alpha-Beta 剪枝的效率。

### 1.2 Alpha-Beta 剪枝 (Alpha-Beta Pruning)
在 Minimax 算法遍历博弈树时，会有大量的分支节点。Alpha-Beta 剪枝通过维护两个值：
- `alpha`：当前搜索路径上，己方（极大方）至少能得到的保证分数。
- `beta`：当前搜索路径上，对手（极小方）最多允许己方得到的分数。

如果在搜索某一个子节点时，发现该节点的评估值使得 `beta <= alpha`，则说明对手绝不会允许局面发展到这个分支，因此可以立即**剪枝（停止搜索该分支）**，从而极大地减少了不必要的计算量。

## 2. 时间估算与控制机制

在复杂局面（例如开局阶段，棋子繁多，单步合法走法可达几十种），搜索树会呈指数级爆炸。为了保证用户体验，UI 层目前设置了全局最大思考时间 **5.0 秒**。引擎是如何严格遵守这个限制的呢？

### 2.1 软性预判提前退出 (Soft Early Exit)
在迭代加深的每一次外层循环开始前，引擎会检查当前已消耗的时间：
```typescript
const elapsed = (Date.now() / 1000) - this.startTime
if (elapsed > this.timeLimit * 0.4 && currentDepth > 1) {
  break
}
```
**原理**：博弈树搜索的时间通常随深度呈指数增长。如果完成前 `N-1` 层已经消耗了总时间限制的 40% 以上，那么计算第 `N` 层极有可能会超时。此时引擎会主动放弃更深层的搜索，直接返回已有结果。

### 2.2 硬性超时阻断 (Hard Timeout)
在遍历每一层级的每一个子节点，以及递归的 `_alphaBeta` 函数内部，引擎会频繁检查当前时间：
```typescript
if ((Date.now() / 1000) - this.startTime > this.timeLimit) {
  timedOut = true
  break // 或者在递归中 return this.evaluator.evaluate(board)
}
```
**原理**：即使在搜索极深层某个极度复杂的分支时，只要耗时超过设定的 `maxTime`，搜索过程会立刻被“暴力”截断。由于有迭代加深的保底，引擎即便在当前深度被截断，依然可以安全地返回上一个完整深度的最优解。

## 3. 计算量优化与启发式策略

仅仅依靠单纯的 Alpha-Beta 剪枝并不足以在几秒内搜索到较高深度。引擎应用了以下几种启发式策略来优化计算效率：

### 3.1 着法排序 (Move Ordering)
Alpha-Beta 剪枝的效率极度依赖于**搜索顺序**。如果最先搜索的是最优步，剪枝发生得最早、最多。
引擎的 `_orderMoves` 方法对生成的合法着法进行了打分和排序，优先计算以下着法：
1. **吃子着法 (Captures)**：
   应用 **MVV-LVA (Most Valuable Victim - Least Valuable Attacker)** 策略。
   公式：`得分 += 目标棋子价值 * 10 - 攻击棋子价值`。
   例如，“小卒吃车”的优先级会远高于“车吃小卒”。
2. **将军着法 (Checks)**：
   如果某步棋走完后能“将军”，额外给予高分（`+500`），促使引擎优先考虑进攻性极强的路线。

通过这种预排序，Alpha-Beta 算法能更快收敛。

### 3.2 静止搜索 (Quiescence Search)
传统的固定深度搜索容易遇到**水平线效应 (Horizon Effect)**。例如，搜索深度为 4，在第 4 步时，引擎发现自己吃掉了对方的车，认为这是一个大优局面；但它没看到在第 5 步对手就会立刻把自己的车吃回来。
为了解决这个问题，当常规的 `_alphaBeta` 搜索达到最大深度 (`depth <= 0`) 时，并不会直接返回静态评估值，而是进入 `_quiescenceSearch`。

**静止搜索的特点**：
- 它只搜索**吃子着法 (Capture Moves)**，直到局面变得“安静”（没有激烈的吃子交换）。
- 为了防止无休止的吃子导致计算量失控，静止搜索内部也有一个绝对深度限制（目前代码中限制为最多往下延伸 4 层）。
- 这个机制大大提高了引擎在战术交换中的准确性，避免了短视的错误。

### 3.3 重复局面惩罚 (Repetition Detection)
为了防止引擎通过无意义的反复将军或重复走子来拖延局势（或者在优势时误走重复步导致判和），搜索树在根节点和 `_alphaBeta` 内部都加入了历史记录校验：
```typescript
if (this.history.some(h => h.startsWith(nextFen))) {
  // 对导致重复的局面给予极大惩罚
  const repetitionPenalty = -500 
  // ...
}
```
这促使引擎在正常情况下积极寻找新的突破口，而不是在原地打转。

---

**总结**：
本地引擎能够根据用户在 UI 上配置的**目标深度（2到4层）**进行探索，但在计算量超出预期时，通过**软硬时间阻断机制**保证能在 5 秒内返回结果；通过**着法排序**和**静止搜索**，确保了在有限的计算资源下，算出来的棋力尽可能精准并具有攻击性。
